# /ac-patch - 创建并执行补丁

用法：`/ac-patch {简短补丁描述}`

该命令基于当前活跃计划上下文执行小步迭代，并在当前计划目录下沉淀补丁文档。

## 严格前置验证（必须全部通过才能继续）

### ✅ 验证 1：必须存在且仅存在一个活跃计划

- 扫描 `.plan/` 下 `plan-{number}` 目录（不包括 `.plan/backlog/` 与 `.plan/done/`）。
- 若数量为 0，终止并提示：

  > ❌ 当前没有可执行计划，请先使用 /ac-plan 创建计划。

- 若数量大于 1，终止并提示：

  > ❌ 检测到多个活跃计划，工作流已异常。请先恢复为单活跃状态。

### ✅ 验证 2：当前计划状态必须为 `已执行`

- 读取当前 `plan.md`。
- 必须包含 `> 状态: 已执行`。
- 若仍为 `未执行`，终止并提示：

  > ❌ 当前计划尚未执行，请先使用 /ac-implement。

## 执行步骤

1. **确定补丁编号**：
   - 扫描当前计划目录下所有 `patch-{number}.md`。
   - 取最大编号 + 1；若不存在则从 1 开始。

2. **加载上下文**：
   - 阅读当前 `plan.md` 的目标、内容、影响范围、历史补丁。
   - 阅读历史补丁文件，避免重复修复。

3. **澄清补丁需求**：
   - 基于用户描述理解补丁目标。
   - 仅在必要时补充澄清问题。

4. **执行补丁并验证**：完成代码修改并验证可构建、可运行、符合预期。

5. **生成补丁文档**：创建 `patch-{编号}.md`：

   ```markdown
   # {补丁名称}

   ## 补丁内容

   {说明修改了什么、为什么修改}

   ## 影响范围

   - 新增文件: `/path/to/file`
   - 修改文件: `/path/to/file`
   - 删除文件: `/path/to/file`
   ```

6. **更新 `plan.md`**：
   - 在 `## 历史补丁` 追加 `- patch-{编号}.md`（避免重复）。
   - 将本次文件变更并入 `## 影响范围`（避免重复）。

## 注意事项

- `/ac-patch` 仅操作当前活跃计划，不直接操作 backlog 计划。
- `/ac-patch` 不改变计划状态，状态保持 `已执行`。
- 补丁应聚焦小步迭代，避免无关改动。
