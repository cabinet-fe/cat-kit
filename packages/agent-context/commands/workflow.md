# 工作流

实现一个基于计划的工作流。

## 全局约束

1. **两阶段模型**：
   - 初始化阶段（仅 `/ac-new`）允许一次创建多个计划。
   - 维护阶段（`/ac-plan`、`/ac-implement`、`/ac-patch`、`/ac-done`）保持单活跃计划模型。
2. **目录语义**：
   - 活跃计划：`.plan/plan-{number}`（最多一个）。
   - 待执行队列：`.plan/backlog/plan-{number}`（可多个）。
   - 已归档计划：`.plan/done/plan-{number}-{date}`。
3. **当前计划定义**：当前计划是 `.plan/` 下唯一的 `plan-{number}`。
4. **两态状态机**：`plan.md` 仅使用 `> 状态: 未执行` 与 `> 状态: 已执行`。
5. **统一编号规则**：新计划编号基于以下位置的最大编号 + 1：
   - `.plan/plan-{number}`
   - `.plan/backlog/plan-{number}`
   - `.plan/done/plan-{number}-{date}`
6. **异常恢复模式**：若 `.plan/` 下存在多个 `plan-{number}`，所有命令必须先拒绝执行，并提示先恢复到单活跃状态。

## 命令

### /ac-new

用于在全新项目中初始化计划系统，可一次创建多个计划。

初始化后的目录结构如下（示例）：

```
.plan/
  done/
  backlog/
    plan-2/
      plan.md
    plan-3/
      plan.md
  plan-1/
    plan.md
```

行为：

- 仅可用于全新项目。
- 若项目已有业务代码或已存在任意计划/归档计划，则不能使用。
- `plan-1` 作为活跃计划，其余计划进入 backlog。

### /ac-plan

用于创建一个新计划。后面可附带简短描述，例如：

```
/ac-plan 我想要新增一个统计分析菜单
```

行为：

- 仅在“无活跃计划且 backlog 为空”时允许执行。
- 根据统一编号规则创建 `.plan/plan-{number}/plan.md`。
- 描述简短时允许多轮澄清。

### /ac-implement

用于实现当前计划（执行完成，不等于目标最终完成）。

行为：

- 仅在存在且仅存在一个活跃计划时允许执行。
- 仅当当前计划状态为 `未执行` 时允许执行。
- 实现完成后将状态改为 `已执行`，并在 `plan.md` 的 `## 影响范围` 记录本次新增/修改/删除文件。

### /ac-patch

用于基于当前计划创建并执行补丁。后面可附带简短描述，例如：

```
/ac-patch 图表样式丑陋，请使用赛博朋克风格重新绘制
```

行为：

- 仅在存在且仅存在一个活跃计划时允许执行。
- 当前计划必须已执行（状态为 `已执行`）。
- 在当前计划目录下生成 `patch-{number}.md`。
- 在 `plan.md` 的 `## 历史补丁` 追加补丁记录，并更新 `## 影响范围`（去重）。

### /ac-done

用于将当前计划标记为真正完成并归档到 `.plan/done`。

行为：

- 仅在存在且仅存在一个活跃计划时允许执行。
- 当前计划状态必须为 `已执行`。
- 必须先得到用户确认“任务已真正完成”。
- 将 `.plan/plan-{number}` 移动到 `.plan/done/plan-{number}-{date}`，其中 `{date}` 格式为 `YYYYMMDD`。
- 归档后若 backlog 非空，自动将 backlog 中最小编号计划晋升到 `.plan/` 作为新的活跃计划。
