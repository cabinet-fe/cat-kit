# Agent Context Skill 协议

本文件定义 `agent-context` Skill 在项目内执行计划生命周期时的统一规则。

## 目标与范围

- 使用单一 `agent-context` Skill 管理计划生命周期。
- 不依赖 `/ac-init`、`/ac-plan` 之类的命令入口。
- 用户通过自然语言表达“初始化 / 计划 / 重规划 / 实施 / 补丁 / 完成”等意图时，Skill 负责匹配到对应动作。

## 生成文件结构

Skill 以「导航 + 子文件」方式组织，`SKILL.md` 为精简导航层，各动作协议独立存放：

```text
{skillRootDir}/agent-context/
├── SKILL.md              # 导航层：意图匹配、全局约束、目录约定
├── actions/
│   ├── init.md
│   ├── plan.md
│   ├── replan.md
│   ├── implement.md
│   ├── patch.md
│   └── done.md
└── agents/
    └── openai.yaml       # 仅 Codex 目标
```

## 全局约束

- 状态机固定为两态：`未执行`、`已执行`。
- 任意时刻最多只能有一个当前计划：`.agent-context/plan-{number}`。
- 若检测到多个当前计划，必须先拒绝执行并提示恢复单活跃状态。
- 计划编号全局递增且不复用。
- 补丁编号在单个计划目录内递增且不复用。

## 生命周期动作

- `init`：初始化项目上下文，可附带描述。
- `plan`：创建新计划，必须具备清晰描述。
- `replan`：重规划未实施计划。
- `implement`：实施当前计划并完成验证循环。
- `patch`：基于当前已实施计划执行补丁。
- `done`：将真正完成的当前计划归档。

## 生命周期与状态约束

- 单次只允许存在一个当前计划目录：`.agent-context/plan-{number}`。
- `preparing/` 存放待执行计划队列。
- `done/` 存放已归档计划。
- `implement` 成功后，当前计划才可标记为 `已执行`。
- `done` 仅用于真正完成后的归档动作。
- `patch` 只允许在当前计划为 `已执行` 后执行。

## 目录与编号规则

```text
project
├── .agent-context/
│   ├── plan-2/
│   │   ├── plan.md
│   │   └── patch-1.md
│   ├── preparing/
│   │   ├── plan-3/
│   │   │   └── plan.md
│   │   └── plan-4/
│   │       └── plan.md
│   └── done/
│       └── plan-1-20260305/
│           ├── plan.md
│           ├── patch-1.md
│           └── patch-2.md
```

- 计划目录命名为 `plan-{number}`，编号严格递增且不可复用。
- 补丁文件命名为 `patch-{number}.md`，在单个计划目录内递增。
- 新计划编号由 `.agent-context/`、`.agent-context/preparing/`、`.agent-context/done/` 的最大计划编号 + 1 决定。
- 归档命名格式固定：`plan-{number}-{YYYYMMDD}`。

## `plan.md` 标准格式

```markdown
# {计划名称}

> 状态: 未执行

## 目标

{明确的目标描述}

## 内容

{详细实施步骤}

## 影响范围

## 历史补丁
```

- 状态行只能是 `> 状态: 未执行` 或 `> 状态: 已执行`。
- 状态行必须唯一且位于标题后。
- `implement` 成功后才可将状态从 `未执行` 更新为 `已执行`。

## 各动作执行规范

### `init`

- 可附加描述用于补充项目背景。
- 新项目：先澄清需求，再生成高质量 `AGENTS.md`，随后进入计划创建流程。
- 旧项目：
  - 根目录不存在 `AGENTS.md` 时，通过提问与代码阅读补全并生成。
  - 已存在 `AGENTS.md` 时，先评估质量，不足再增补优化。
- 区分原则：新项目会创建计划；旧项目默认不创建计划。

### `plan`

- 执行前需确保：
  - 不存在进行中的 `preparing` 计划冲突。
  - 不存在已实施但未归档的当前计划。
  - 不存在多个当前计划。
- 需求存在歧义时必须先向用户澄清；无歧义则直接执行。
- 计划按复杂度可拆分多个：
  - 多计划：最小编号放入 `.agent-context/` 作为当前计划，其余放入 `.agent-context/preparing/`。
  - 单计划：直接放入 `.agent-context/`。
- 生成计划后需自检：步骤可独立执行、粒度适当、影响范围可预估；不通过则修改后重新检查。

### `replan`

- 前提是至少存在一个未实施计划。
- 默认作用域为 `.agent-context/preparing/` 中全部未实施计划；用户明确要求时，可将当前未实施计划纳入重规划范围。
- 若当前计划为 `已执行`，禁止重写当前计划内容；仅允许重规划 `preparing/` 队列。
- 重规划后必须保持“单当前计划 + 若干 preparing 计划”的结构不变。

### `implement`

- 用于实施当前计划。
- 实施完成后进入验证循环：逐项对照计划内容确认完整性，运行项目验证（类型检查 -> lint -> 测试），失败则修复后重新验证。
- 全部通过后才可更新状态为 `已执行`。

### `patch`

- 必须附带补丁描述。
- 每次补丁需创建并记录 `patch-{number}.md`。
- 执行后需更新 `plan.md` 的 `## 影响范围`，并进行去重。

### `done`

- 仅用于将当前已执行且真正完成的计划归档到 `.agent-context/done/`。
- 归档目录命名为 `plan-{number}-{date}`，其中 `{date}` 格式必须为 `YYYYMMDD`。
- 归档后若 `preparing/` 非空，应自动晋升最小编号计划为新的当前计划。

## 高质量 `AGENTS.md` 标准

- 常用约束前置，优先降低代理执行歧义。
- 明确目录结构、技术栈与代码风格约束。
- 内容精简，避免冗余上下文。
- 大型单体仓库可按子包拆分维护本地 `AGENTS.md`。
