# Agent Context 执行指南

本文件定义代理在项目内执行 Agent Context 工作流时的统一规则。

## 目标与范围

- 使用一组带 `ac` 前缀的命令/工作流管理计划生命周期。
- 命令命名可能因工具差异表现为 `ac:` 或 `ac-` 前缀。
- 本指南聚焦行为约束与目录语义，不包含具体业务实现细节。

## 全局约束

- 状态机固定为两态：`未执行`、`已执行`。
- 任意时刻最多只能有一个当前计划：`.agent-context/plan-{number}`。
- 若检测到多个当前计划，所有命令必须先拒绝执行并提示恢复单活跃状态。
- 计划编号全局递增且不复用。
- 补丁编号在单个计划目录内递增且不复用。

## 命令集合

- `init`：初始化项目上下文（可附带描述）。
- `plan`：创建新计划（必须附带描述）。
- `replan`：重规划已有未实施计划（必须附带描述）。
- `implement`：实施当前计划（不可附带描述）。
- `patch`：基于当前计划执行补丁（必须附带描述）。
- `done`：将当前计划标记为真正完成并归档（不可附带描述）。

## 生命周期与状态约束

- 单次只允许存在一个“当前计划”目录：`.agent-context/plan-{number}`。
- `preparing` 用于存放待执行计划队列。
- `done` 用于存放已归档计划。
- `implement` 完成后，当前计划应被标记为“已实施”。
- `done` 仅用于“真正完成”后的归档动作，归档后目录命名为 `plan-{number}-{YYYYMMDD}`。
- `patch` 只允许在当前计划为“已实施”后执行。

## 目录与编号规则

目录结构示例：

```text
project
├── .agent-context/
│   ├── plan-2/
│   │   ├── plan.md
│   │   └── patch-1.md
│   ├── preparing/
│   │   ├── plan-3/
│   │   │   └── plan.md
│   │   └── plan-4/
│   │       └── plan.md
│   └── done/
│       └── plan-1-20260305/
│           ├── plan.md
│           ├── patch-1.md
│           └── patch-2.md
```

统一规则：

- 计划目录命名为 `plan-{number}`，编号严格递增且不可复用。
- 补丁文件命名为 `patch-{number}.md`，在单个计划目录内递增。
- 新计划编号由以下范围共同决定：`.agent-context/`、`.agent-context/preparing/`、`.agent-context/done/`，取最大编号 + 1。
- 归档命名格式固定：`plan-{number}-{YYYYMMDD}`。

## `plan.md` 标准格式

每个计划目录中的 `plan.md` 必须遵循以下最小结构：

```markdown
# {计划名称}

> 状态: 未执行

## 目标

{明确的目标描述}

## 内容

{详细实施步骤}

## 影响范围

## 历史补丁
```

状态字段规则：

- 仅允许 `> 状态: 未执行` 或 `> 状态: 已执行`。
- 状态行必须唯一且位于标题后，禁止使用其他别名（如“进行中”“完成”）。
- `implement` 成功后才可将状态从 `未执行` 更新为 `已执行`。

## 各命令执行规范

### `init`

- 可附加描述用于补充项目背景。
- 新项目：先澄清需求，再生成高质量 `AGENTS.md`，随后进入计划创建流程（等价继续执行 `plan`）。
- 旧项目：
  - 若根目录不存在 `AGENTS.md`，需通过提问与代码阅读补全并生成。
  - 若已存在 `AGENTS.md`，需先评估质量，不足时再增补优化。
- 区分原则：新项目会创建计划；旧项目默认不创建计划。

### `plan`

- 必须附带描述。
- 执行前需确保：
  - 不存在进行中的 `preparing` 计划。
  - 不存在已实施但未归档的当前计划。
- 计划按复杂度可拆分多个：
  - 多计划：最小编号放入 `.agent-context/` 作为当前计划，其余放入 `.agent-context/preparing/`。
  - 单计划：直接放入 `.agent-context/`。
- 必须有机制可判断 `.agent-context/plan-{number}` 是否已执行。

### `replan`

- 必须附带描述。
- 前提是至少存在一个未实施计划（位于 `preparing` 或当前未实施计划）。
- 默认作用域为 `.agent-context/preparing` 中全部未实施计划；可通过描述指定仅重规划部分计划。
- 若当前计划为 `已执行`，禁止重写当前计划内容；仅允许重规划 `preparing` 队列。
- 若当前计划为 `未执行` 且用户明确要求，可将当前计划纳入重规划范围。
- 重规划后必须保持“单当前计划 + 若干 preparing 计划”的结构不变。
- 新增计划编号必须继续使用全局最大编号 + 1，已有未改动计划保持原编号。
- 冲突处理：
  - 目标计划不存在：拒绝执行并列出可选计划编号。
  - 目标计划已执行：拒绝执行并提示使用 `patch`。
  - 检测到多个当前计划：拒绝执行并提示先恢复单活跃状态。

### `implement`

- 不可附带描述。
- 用于实施当前计划。
- 实施完成后必须更新当前计划“已实施”状态；该状态用于控制后续命令可执行性。

### `patch`

- 必须附带描述。
- 仅允许在当前计划已实施后执行。
- 每次补丁需创建并记录 `patch-{number}.md`。
- 执行后需更新 `plan.md` 的 `## 影响范围`，并进行去重。

### `done`

- 不可附带描述。
- 仅用于将当前计划标记为真正完成并归档到 `.agent-context/done`。
- 归档目录命名为 `plan-{number}-{date}`，其中 `{date}` 格式必须为 `YYYYMMDD`。
- 归档后若 `preparing` 非空，应自动晋升最小编号计划为新的当前计划。

## 高质量 `AGENTS.md` 标准

- 常用命令与约束前置，优先降低代理执行歧义。
- 明确目录结构、技术栈与代码风格约束。
- 内容精简，避免冗余上下文。
- 大型单体仓库可按子包拆分维护本地 `AGENTS.md`。

## CLI 工具

- `setup`：交互式生成工作流命令。
- `update`：批量更新工作流命令（仅针对已经存在的工具）。
