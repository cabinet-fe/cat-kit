# 代理上下文框架

## 需求

### 命令/工作流

大部分AI变成工具具备命令功能，它实际上是一个个特点位置特定名称的 markdown 文件。

所有的命令都冠以`ac`前缀，有的工具中可能为`ac:`, 有的工具可能为`ac-`。

本框架提供一系列命令：

- `init` 初始化项目
- `plan` 创建新计划
- `replan` 重新规划当前计划
- `implement` 实施当前计划
- `patch` 基于当前计划创建并执行补丁
- `done` 将当前计划标记为真正完成并归档

#### init 命令

该命令后面可以附带描述。

如果是全新项目：

1. 它会引导你澄清需求，然后帮助你生成一个高效的`AGENTS.md`文件再生成一系列计划，**生成计划实际上就是接着执行`plan`命令**。

如果是旧项目：

1. 如果你的项目根目录下不存在`AGENTS.md`文件，它会向你询问这个项目的一些基础情况或者你在命令中明确告诉它一些信息，然后代理会自行阅读你的代码并辅助你生成一个高效的`AGENTS.md`文件。

2. 如果`AGENTS.md`文件存在，它会读取`AGENTS.md`文件， 并判断这是否是一个好的`AGENTS.md`文件， 如果这个文件不够完善，它会帮助你完善这个文件。

新项目和旧的项目有个很明显的区别：新项目创建计划，旧的项目不会创建计划。

一个好的`AGENTS.md`文件应该具备以下特质：

- 将项目常用的命令或者指令放到靠前的位置
- 能够清晰描述项目目录结构（尤其是对现存的项目来说）
- 明确的技术栈、代码风格
- 不要臃肿，只保留必要的内容以便于节省上下文
- 对于大型单体仓库，可以将其分散到子包中去

#### plan 命令

该命令后面必须附带描述。该命令需要确保当前不存在`preparing`计划，不存在已实施的计划，否则告诉用户接下来应该继续实施当前计划（调用某些命令等）。

它的职责是引导澄清需求（如果你的需求不够明确的话），创建计划，计划的多少与需求的具体的复杂度相关。计划是一系列格式为`plan-{number}`的目录，其中包含`plan.md`文件，`patch-{number}.md`文件。编号序号需要严格递增，先去`done`目录下找目录序号，确定当前最新序号。

计划创建后， 有两种情况：

1. 如果生成了多个计划，编号小的那个直接移动到`.agent-context`目录下，其余的移动到`.agent-context/preparing`目录下
2. 如果生成了单个计划，直接移动到`.agent-context`目录下

注意：需要有明确的机制来确保`.agent-context`目录下的`plan-{number}`目录是否已执行。

#### implement 命令

该命令后面**不要**附带描述。

该命令用于实施当前计划，计划实施完成后，标记此次实施完成（这个是否完成状态很重要，一些命令是否可以执行取决于它）。

#### patch 命令

该命令后面必须附带描述。需确保当前计划实施后才能执行。

该命令用于基于当前计划创建并执行补丁。执行后添加补丁记录`patch-{number}.md`文件，并更新`plan.md`的`## 影响范围`（去重）。

#### done 命令

该命令后面**不要**附带描述。

该命令用于将当前计划标记为真正完成并归档到`.agent-context/done`。归档后的计划目录格式为`plan-{number}-{date}`，其中`{date}`格式为`YYYYMMDD`。

#### 上下文目录示例

```
project
├── .agent-context/    # 代理上下文目录
│   ├── plan-2         # 计划2
│   │   ├── plan.md    # 计划文件
│   │   └── patch-1.md # 补丁1
│   ├── preparing/       # 准备中的计划
│   │   ├── plan-3     # 准备中的计划3
│   │   │   ├── plan.md    # 计划文件
│   │   └── plan-4     # 准备中的计划4
│   │       ├── plan.md    # 计划文件
│   ├── done/          # 已完成计划目录
│   │   ├── plan-1     # 已完成计划1
│   │   │   ├── plan.md    # 计划文件
│   │   │   ├── patch-1.md # 补丁1
│   │   │   └── patch-2.md # 补丁2
```

### 命令行工具

命令名称: `agent-ctx`

子命令:

- `setup`: 交互式生成代理框架的代理命令/工作流、SKILLS。
- `update`: 更新所有代理命令/工作流、SKILLS

注：交互式主要是利用`@inquirer/prompts`来实现的，目前主要用来指定工具类别。

### SKILLS

SKILLS主要是对代理`命令/工作流`的补充，减少代理的上下文损耗，比如可以内置一些脚本做一些验证工作等等，具体有哪些还没有想好。

## 工具支持

框架支持以下工具

- Claude Code
- Codex
- Cursor
- Antigravity
- Github Copilot

不同的工具生成的代理命令/工作流、SKILLS的目录和结构不同，主要是这些区别
