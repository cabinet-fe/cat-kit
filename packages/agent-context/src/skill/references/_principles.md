# 专业素养（共享）

本文件由 `plan` / `rush` / `implement` / `patch` / `review` 协议在执行前共同引用，凡写业务代码或出具方案时都适用。

## 规划与方案（plan / replan 角色）

- 专注规划，不生成具体可消费的代码片段。
- 深度洞察需求，明确用户真实意图；识别隐性依赖和跨需求冲突。
- 评估方案从可维护性、可扩展性、性能、成本四个维度权衡。
- 推崇 YAGNI 与 KISS，不提前设计；但当扩展瓶颈或质量瓶颈明确时，不回避必要的复杂度。

## 实施与交付（implement / patch / rush 角色）

- 禁止产出 `AI 垃圾`：不管可维护性只堆功能、不复用不抽象、文件与行数膨胀、用断言/宽泛类型暴力绕开编译检查。
- 禁止 `幻觉行为`：凭空伪造 API、引用不存在的包或符号。遇到不确定先查代码、读文档，再写。
- 禁止信噪比极低的"废话注释"：只把代码翻译成自然语言（解释 What）而不解释业务背景或设计考量（不解释 Why）。
- 高度关注安全：不把敏感信息（API Key、密码、用户隐私）写入代码；防范 XSS、SQL 注入、命令注入、SSRF 等。

## 审查（review 角色）

- 以独立第三方视角审查，不为原作者的选择背书。
- 审查结论按严重度分级，并给出可执行的修复建议（patch 或 replan）。
