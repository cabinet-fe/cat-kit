# /ac-done

## 严格前置验证

- 不可附带描述，带描述时拒绝执行。
- 必须存在且仅存在一个当前计划（`.agent-context/plan-{number}`）。
- 当前计划状态必须为 `已执行`（状态为 `未执行` 时拒绝执行，提示先运行 `ac-implement`）。
- 必须得到用户确认"任务已真正完成"后才可执行归档。

## 执行步骤

1. **用户确认**：向用户确认当前计划已真正完成，可以归档。
2. **归档当前计划**：
   - 将 `.agent-context/plan-{number}` 移动到 `.agent-context/done/plan-{number}-{YYYYMMDD}`。
   - `{YYYYMMDD}` 为归档当日日期，格式固定（如 `20260306`）。
   - 归档目录包含完整的 `plan.md` 与所有 `patch-{number}.md`。
3. **晋升 preparing 队列**：
   - 检查 `.agent-context/preparing/` 是否非空。
   - 若非空：将最小编号的计划目录移动到 `.agent-context/` 作为新的当前计划。
   - 若为空：无操作，此时无当前计划。
4. **校验结果**：
   - 确认归档目录存在且完整。
   - 确认晋升后仍满足"最多一个当前计划"约束。

## 归档命名规则

- 格式固定为 `plan-{number}-{YYYYMMDD}`。
- `{number}` 为原计划编号，不可修改。
- `{YYYYMMDD}` 为执行 `ac-done` 当日的日期。
- 归档后计划编号不可被新计划复用。

## 输出要求

完成后向用户反馈：

- 已归档的计划编号与归档路径。
- 是否晋升了 preparing 计划，若是则告知晋升的计划编号。
- 当前是否还有待执行计划。

## 失败条件

- 附带了描述 → 拒绝执行。
- 当前计划不存在 → 拒绝执行，提示先运行 `ac-plan`。
- 当前计划状态为 `未执行` → 拒绝执行，提示先运行 `ac-implement`。
- 用户未确认 → 中止执行，不归档。
- 存在多个当前计划 → 拒绝执行，提示恢复单活跃状态。
- 归档后校验失败（目录不完整或晋升后出现多个当前计划）→ 回滚并报错。
