# 用户提问规范

本文件规定在协议执行中如何向用户发起结构化的交互式提问。**协议里每一处"通过交互式提问工具..."的指令都按本文件执行**，不要凭名字直觉判断工具是否可用。

## 工具识别（首次提问前必做）

"交互式提问工具"不是固定的某个名字，而是**宿主 / 运行时提供的、能发起单选 / 多选 / 带选项文本询问并暂停等待用户回复的工具集合**。不同 host 命名完全不同，按以下步骤识别当前环境是否提供该能力：

1. 扫描本次会话中**你自己当前可调用**的工具清单（工具名 + 工具描述）。
2. 若**任一**工具的名字或描述命中下列语义之一，即视为命中：
   - 向用户提问 / 让用户从选项中选择 / 收集用户输入 / 多选题 / 单选题 / 请求用户确认或澄清
   - 英文关键字参考：`ask`、`question`、`input`、`prompt`、`choice`、`select`、`multiple_choice`、`followup`、`clarify`、`user`
3. 命中任一 → **必须调用该工具**发起提问。**不允许**以"优先"、"可选"、"我这里不方便"等理由跳过，也不允许改用普通文本回复代替。
4. 真的**完全没有**任何命中 → 才用一条简短文本问题直接询问用户并暂停本轮，**不要伪造工具调用**，不要在没等到答复的前提下继续推进。

**不确定是否命中时倾向调用**：错误调用代价小（用户可追问或忽略），漏问导致的决策错误代价大。

**已知工具名示例（不穷举，仅辅助识别）**：

| 宿主 / 来源 | 工具名 |
| --- | --- |
| Cursor | `AskQuestion` |
| Claude Code | `AskUserQuestion` |
| Codex CLI | `request_user_input` / `RequestUserInput` |
| Cline / Roo | `ask_followup_question` |
| 通用 / 其他 | `Question` / `askQuestions` / `question` / `prompt_user` |

遇到上表未覆盖的新 host，仍按第 1–4 步的**语义规则**判断，不要因"名字没见过"就回退到纯文本。

---

**任何情况下调用交互式提问工具时必须遵守以下规范**：

## 何时禁止提问（红线）

命中任一条就直接跳过提问，按默认/推荐路径继续：

- 用户描述已给出明确答案或可推断的唯一合理路径。
- 两个候选选项没有实质差异（例如仅"继续 / 取消"）——直接继续。
- 本轮对话中已就同一主题追问过，且用户未提供新信息。
- 单次协议执行中累计交互式提问已达 3 次，且后续问题并非阻塞性决策——改为执行完再向用户汇总。
- 问题本质是"代理自己可查"的信息（例如文件是否存在、类型定义、依赖版本），必须先自行检查再提问。

## 基础规范

- 提问要通俗易懂，不要废话，禁止拽花哨、华而不实的文风，给出的选项同理。
- 单选提问在存在客观更优解时必须在问题末尾标注推荐项并说明理由；不存在可合理判断的更优解时（例如纯偏好），可不标推荐，但要在理由位置写明"无默认倾向"。
- 一次提问聚焦一个主题，不在单个问题中混杂多个无关决策。

## 反向追问

信息收集和需求澄清环节，不能只被动接收用户说法——必须主动审视用户的陈述，挖掘未被意识到的盲区。

**探测方向**：

- **假设挑战**：用户描述中哪些隐含假设可能不成立
- **边界探测**：哪些边界/异常场景未被覆盖
- **遗漏发现**：落地所需的关键决策中哪些还没做
- **矛盾检测**：需求之间或需求与现有系统之间是否冲突
- **风险预警**：技术可行性、性能瓶颈、维护成本等隐患

**执行要求**：

- 仅当识别到至少一个**显著盲区**时才发起反向追问；未发现时不要为了"走流程"而强制提问。
- 每轮聚焦 2-3 个最有价值的问题，不一次倾泻所有疑问
- 每个问题附带简短理由——说明为什么这个问题值得关注
- 连续一轮追问未发现新的显著盲区 → 结束
- 最多追问 2 轮，除非用户主动要求继续
