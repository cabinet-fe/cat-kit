# /ac-implement

## 严格前置验证

- 不可附带描述，带描述时拒绝执行。
- 必须存在且仅存在一个当前计划（`.agent-context/plan-{number}`)。
- 当前计划状态必须为 `未执行`（已执行时拒绝，提示使用 `ac-patch` 或 `ac-done`)。
- `plan.md` 的 `## 目标` 与 `## 内容` 必须有实质内容（空内容时拒绝执行）。

## 执行步骤

1. 读取当前计划的 `plan.md`，理解 `## 目标` 与 `## 内容`。
2. 依据 `## 内容` 中的步骤逐项实施。
3. 完成必要验证（测试、类型检查、lint 等）。
4. 更新 `plan.md` 的状态行：

   将：
   ```
   > 状态: 未执行
   ```
   更新为：
   ```
   > 状态: 已执行
   ```

5. 更新 `plan.md` 的 `## 影响范围`，按以下格式记录所有变更文件：

   ```markdown
   ## 影响范围

   - 新增文件: `/path/to/new-file`
   - 修改文件: `/path/to/modified-file`
   - 删除文件: `/path/to/deleted-file`
   ```

## 状态更新规则

- 状态更新必须发生在实施完成且验证通过之后，不可提前更新。
- 状态行必须唯一（整个 `plan.md` 中只能出现一次 `> 状态:` 行）。
- 仅允许从 `未执行` 更新为 `已执行`，不可使用其他状态值。

## 约束

- `ac-implement` 仅操作当前计划，不直接操作 `preparing/` 中的计划。
- 实施过程中若遇到阻塞问题，应向用户报告而非静默跳过。

## 失败条件

- 附带了描述 → 拒绝执行。
- 当前计划不存在 → 拒绝执行，提示先运行 `ac-plan`。
- 当前计划状态为 `已执行` → 拒绝执行，提示使用 `ac-patch` 或 `ac-done`。
- `## 目标` 或 `## 内容` 为空 → 拒绝执行，提示补充计划内容。
- 存在多个当前计划 → 拒绝执行，提示恢复单活跃状态。

## 完成提示

执行完毕后，必须使用以下固定格式提示用户下一步操作：

> **下一步建议：**
> - `/ac-done`：归档当前计划（确认完成时）
> - `/ac-patch`：执行补丁修复（发现问题时）
