# replan

重新规划已有的未实施计划，保持「单当前计划 + preparing 队列」结构不变。

> **前提**：此协议的所有路径和编号均来自上下文脚本输出，禁止自行扫描目录计算编号。

必须附带重规划描述。

## 核心原则（替换而非新开）

- **原地替换**：凡纳入本协议作用域的未执行计划，均在**既有** `plan-{N}/` 目录内重写 `plan.md`（及按需增删该目录下的 `patch-*.md`），**保留原编号 N 与目录路径**。
- **禁止**：将状态仍为「未执行」的计划移入 `.agent-context/{scope}/done/`，或删除后以 `nextPlanNumber` 新建目录来「重做同一份计划」（会导致编号跳变，且与「重规划」语义不符）。
- `nextPlanNumber` **仅用于**重规划后**净新增**的未执行计划目录（例如拆分时多出来的独立计划）；**不得**用来替代仍应作为当前计划或 preparing 条目的既有 `plan-{N}/`。

## 前置检查

- 描述为空 → 向用户获取重规划描述后继续执行。
- `currentPlanStatus` 为 `null` 且 preparing 队列为空 → 通过 AskUserQuestion 提供选项：1) 创建新计划 2) 终止操作，按用户选择执行。

## 作用域

- 固定覆盖**全部**未执行计划，不支持部分重规划。
- 未执行计划包括：当前计划（当且仅当其 `currentPlanStatus` 为 `"未执行"`）+ `.agent-context/{scope}/preparing/` 中全部计划。
- `currentPlanStatus` 为 `"已执行"` → 当前计划不纳入重规划；仅对 `preparing/` 队列中的计划进行重规划。

## 执行步骤

1. 结合描述确定重规划方向。若重规划方向仍不明确（拆分方式、优先级调整、范围增减等），通过 AskUserQuestion 提问澄清后再继续。
2. **反向面试**：分析原计划需要调整的根本原因，评估新方向是否真正解决了问题。关注：重规划的核心驱动因素（技术障碍 / 需求变更 / 设计缺陷 / 优先级调整）、新方案是否引入原计划不存在的风险或约束。通过 AskUserQuestion 提出关键探测问题。若分析后无显著问题可跳过。
3. 读取上述作用域内**所有**未执行计划的 `plan.md`，理解现有意图。
4. 生成新的拆分方案，保持「单当前计划 + 若干 preparing 计划」结构；在**既有目录上**落盘，不要为了「干净」而整体迁到全新编号。
5. **编号与目录**：对步骤 3 中涉及的每个既有 `plan-{N}/`，**复用 N**，更新其 `plan.md`（状态保持 `未执行`）。若有净新增的未执行目录，自 `nextPlanNumber` 起递增分配编号。若合并导致未执行目录变少，删除或合并多余目录，且**不得**把仍为「未执行」的计划移入 `done/`。
6. 确保每个未执行计划的 `plan.md` 遵循标准模板。
